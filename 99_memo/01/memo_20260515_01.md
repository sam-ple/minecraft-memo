## ■ Ping/TPS/FPS

| 項目   | 意味                 | 良い状態の目安   | 悪化すると起きること        |
| ---- | ------------------ | --------- | ----------------- |
| Ping | 通信の遅延時間（自分 ↔ サーバー） | 0〜50ms    | 攻撃遅延、位置ズレ、判定ズレ    |
| TPS  | サーバー処理速度           | 20TPS     | 全員ラグ、Mob停止、コマンド遅延 |
| FPS  | 自分の画面描画速度          | 60FPS以上推奨 | 画面カクカク、視点重い       |

---

# Ping（通信遅延）

> 自分とサーバーの「距離・通信速度」

単位は ms（ミリ秒）。

低いほど快適。

## Ping目安

| Ping      | 体感       |
| --------- | -------- |
| 0〜20ms    | 超快適      |
| 20〜50ms   | 快適       |
| 50〜80ms   | 普通       |
| 80〜120ms  | 少し違和感    |
| 120〜180ms | PvPかなり不利 |
| 180〜250ms | 判定ズレ大きい  |
| 250ms〜    | かなり厳しい   |

## Ping悪化で起きること

* 攻撃判定遅延
* execute位置ズレ
* damage eventズレ
* ノックバック違和感
* 相手がワープ気味

※ 特に海外サーバーで起きやすい。

---

# TPS（サーバー処理速度）

> サーバーが1秒間に何tick処理できるか

マイクラ理想値：

20\ \mathrm{TPS}

## TPS目安

| TPS   | 状態       |
| ----- | -------- |
| 20    | 理想       |
| 18〜20 | 普通       |
| 15〜18 | 少し重い     |
| 10〜15 | 明確にラグ    |
| 5〜10  | 危険       |
| 5未満   | ゲーム崩壊レベル |

## TPS低下で起きること

* 全員カクカク
* Mob停止
* execute遅延
* レッドストーン遅延
* アイテム拾えない
* damage event遅れる

## TPSに影響するもの

* execute大量
* entity大量
* MOD/Plugin大量
* Mob大量
* CPU性能不足

---

# FPS（画面描画速度）

> 自分のPCが1秒間に何回画面更新できるか

高いほど滑らか。

## FPS目安

| FPS       | 体感     |
| --------- | ------ |
| 144FPS〜   | 非常に滑らか |
| 60〜144FPS | 快適     |
| 30〜60FPS  | 普通     |
| 20〜30FPS  | カクつく   |
| 20FPS未満   | かなり重い  |

## FPS低下で起きること

* 視点移動カクカク
* 描画遅延
* チャンク表示遅い
* UI重い

※ 基本的に「自分だけ」起きる。

## FPSに影響するもの

* GPU（グラボ）
* シェーダー
* 描画距離
* MOD
* 解像度

```
from time import sleep

from java import JavaClass

from mc_tools import get_tps
from minescript_plus import Hud, Server

mc = JavaClass("net.minecraft.client.Minecraft").getInstance()
Integer = JavaClass("java.lang.Integer")


def main():
    fps_text_id = Hud.add_text("-- FPS", 2, 2)
    ping_text_id = Hud.add_text("-- MS", 2, 14)
    tps_text_id = Hud.add_text("-- TPS", 2, 26)

    try:
        while True:
            fps = mc.getFps()
            ping = Server.get_ping()
            tps = round(get_tps())

            if tps < 20:
                tps_color = "§6"
            elif tps < 15:
                tps_color = "§c"
            else:
                tps_color = "§f"

            Hud.set_text_string(fps_text_id, f"§l{fps} §fFPS")
            Hud.set_text_string(ping_text_id, f"§l{ping} §fMS")
            Hud.set_text_string(tps_text_id, f"{tps_color}§l{tps} §fTPS")

            sleep(0.02)

    except KeyboardInterrupt:
        Hud.remove_text(fps_text_id)
        Hud.remove_text(tps_text_id)
        Hud.remove_text(ping_text_id)


if __name__ == "__main__":
    main()
```

# かなり正しいポイント

## 1. 「RAMよりCPU/TPS」が重要

これは本当にその通り。

Minecraft Java は基本：

```text
TPS低下 ≒ CPU処理不足
```

になりやすい。

特にあなたのような：

* execute
* entity検索
* damage判定
* scoreboard
* command chain

系はシングルスレッド寄り負荷が強い。

なので：

| 強い       | 弱い      |
| -------- | ------- |
| 高クロックCPU | RAM盛るだけ |

になりやすい。

M1/M2/M3 や Ryzen 7000/9000 系を推してるのもかなり理にかなってる。

---

# 2. `entities()` 毎tick危険

これもかなり重要。

特に MineScript 側の：

```python
entities()
```

は単なる「取得」ではなく、

* サーバー同期
* packet受信
* Python object化
* JSON化
* Pythonループ

まで入る。

だから datapack の：

```mcfunction
execute as @e
```

より重いケースも普通にある。

ここを理解してるのはかなり大きい。

---

# 3. damage event 中心は正解

これは PvP/鬼ごっこ系ではかなり強い。

例えば：

## 悪い例

毎tick：

```mcfunction
distance=..2
```

で近接監視。

↓

人数増加で explode。

---

## 良い例

```text
攻撃発生時だけ処理
```

つまり：

* event駆動
* 必要時のみ

にしてる。

これはかなり正しい。

---

# 4. `@e` と `@a` の違いをちゃんと意識できてる

これはかなり重要。

例えば：

## 危険

```mcfunction
execute as @e
```

---

## 安全

```mcfunction
execute as @a[tag=playing]
```

Entity数は本当にTPSへ効く。

特に：

* item
* arrow
* armor_stand
* dropped XP
* mob

全部含まれるので。
